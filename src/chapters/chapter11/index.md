# 第11章：メトリクス設計（Lead time / Review time / 差し戻し率）

## この章で扱うこと

- 計測対象（Lead time / Review time / Reopen 率 等）
- 計測→改善サイクルを運用手順として定義する

## メトリクス設計の目的

エージェント導入は「速くなった気がする」で終わりがちです。
メトリクスは、改善を **再現可能な運用** に落とすための仕組みです。

## 計測対象（例）

最低限、次のような指標を定義します（定義と計測方法のセットで持つ）。

- Lead time：Issue 作成（または着手）からマージまで
- Review time：PR 作成から承認まで
- Reopen 率：クローズ後の再オープン比率（仕様不足/品質不足のシグナル）
- 差し戻し率：レビューでの大きな手戻り（差分肥大/テスト不足のシグナル）

### AgentOps 固有の品質指標

AI agent の導入効果は、速度だけで評価すると危険です。次の指標をセットで見ると、
「速くなったが review / security / eval が弱くなった」状態を検知しやすくなります。

| 指標 | 定義例 | 悪化時に見る原因 |
| --- | --- | --- |
| review 完了率 | review 本文・inline comment・suggestion の全件対応率 | PR 分割、指摘分類、対応証跡の不足 |
| unresolved thread 残存率 | merge 直前に未解決 thread が残った比率 | 完了ゲートの形骸化、レビュー運用の不足 |
| post-merge 失敗率 | merge 後 main checks / Pages / release が失敗した比率 | PR 内検証の不足、環境差分、required checks 漏れ |
| eval 再実行率 | agent / prompt / tool 変更後に評価を再実行した比率 | AI testing strategy との接続不足 |
| 外部投入例外率 | AI/外部サービス投入で承認例外が発生した比率 | 情報分類、マスク、provider 条件確認の不足 |
| コスト逸脱率 | task / PR / eval 単位の上限を超過した比率 | モデル/effort 選択、差分肥大、再試行増加 |

eval は「AI が正しく動いたか」を見るだけでなく、運用変更の安全確認にも使います。
prompt、tool、MCP / tool exposure、権限、モデル設定を変えた場合は、回帰検知の対象として記録します。
テスト戦略との接続は、AI テスト戦略の書籍で扱う eval / regression / golden set の用語に合わせます。

## 収集方法（最小）

重要なのは「完璧な計測」より **継続可能** な収集です。

- GitHub の Issue/PR メタデータを使う（作成日時、マージ日時、レビュー、ラベル等）
- 定義のブレを抑える（ラベル運用、テンプレ必須項目、DoD の固定）
- サンプリング（重要リポジトリ/重要ラベル）から開始し、対象を広げる

## 改善サイクル（運用手順）

1. 収集（週次/月次など頻度を固定）
2. 解釈（どの指標が悪化し、なぜか仮説を立てる）
3. 施策（テンプレ改善、policy / control surface 調整、CI 強化、Skill 追加など）
4. 検証（次サイクルで効果を確認し、記録する）

メトリクスは単独最適化すると副作用が出るため、複数指標と定性レビューをセットで運用します。

## 導入チェックリスト（ドラフト）

- [ ] 計測項目と収集方法（データソース/定義）が定義されている
- [ ] 計測の前提（ラベル運用、テンプレ必須項目、DoD）が整備されている
- [ ] review 完了率、unresolved thread 残存率、post-merge 失敗率を追える

## 運用チェックリスト（ドラフト）

- [ ] 改善サイクル（頻度/担当/レビュー観点）が運用できている
- [ ] 施策→効果検証→記録が継続できている（指標と定性の両方）
- [ ] agent / prompt / tool / MCP / tool exposure / モデル設定の変更時に eval 再実行要否を判断している
